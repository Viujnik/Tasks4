# TaskZ: Валидаторы и очереди

Добро пожаловать в систему управления задачами, где каждая строка данных проверяется, каждая задача имеет свой характер, а очередь умеет больше, чем просто хранить элементы.

## Архитектурные слои

### Валидаторы (src/validators)

+ **fields.py** – дескриптор `ValidatedField`, который не пускает грязь в святая святых. Любое присваивание проверяется набором правил.
+ **rules.py** – коллекция правил: `TypeRule`, `MinLenRule`, `MaxLenRule`, `MinValueRule`, `MaxValueRule`, `StatusRule`.
  Все они следуют протоколу `Rule` – достаточно реализовать метод `validator(field, value)`.
+ **errors.py** – свои исключения `LenError` и `StatusError`.

### Модель задачи (src/tasks_models.py)

Класс `Task` – центральная сущность. Атрибуты защищены дескрипторами:

+ `task_id` (int, положительный)
+ `task_type` (str, 3-16 символов)
+ `description` (str, 8–128 символов)
+ `priority` (int, 1–5)
+ `status` (один из: created, in_progress, in_review, finished)
+ `created_at` (datetime, присваивается автоматически)
+ `deadline` (datetime)
+ `payload` (dict)

**Вычисляемые свойства** (non-data descriptors):

+ `is_on_time` – `True`, если задача завершена (`finished`) и дедлайн не просрочен.
+ `summary` – короткая строка: `Task #123 (type) | status | priority | deadline`

### Источники задач (src/sources)

Три способа создать задачу:

+ **FileSource** – имитация файловой системы. Генерирует задачу с рандомным файлом (имя, размер, отправитель/получатель).
+ **ConsoleSource** – спрашивает у пользователя команду и случайный статус выполнения (OK/ERROR/WARNING/CRITICAL). С помощью yield команда запрашивается только в момент создания.
+ **APISource** – имитация REST API. Метод случайный (GET/POST/PUT/PATCH), URL зашит как `https://rkn.gov.ru`, а статус ответа всегда `ERROR` – жизненно.

Каждый источник реализует `get_task() -> Task` и `printf_task(task) -> str` для красивой печати.

### Очередь задач (src/queue.py)

`TaskQueue` внутри использует `deque` и реализует:

+ `add_task(task)` – в конец.
+ `pop_left()` – забрать слева и удалить.
+ `__len__` и `__iter__` – можно использовать в циклах.
+ `__getitem__` - получение задачи из очереди по индексу
+ `filter(condition)` – ленивый генератор.
+ `filter_by_status(status)`, `filter_by_priority(priority)`, `filter_by_deadline()` – фильтрация задач по полю.
+ `find_by_id(task_id)` – поиск по ID.

Очередь глобальна в модуле `tasks_functions` и живёт, пока вы не закроете программу.

## Интерактивная консоль (src/main.py)

Запускается скрипт, который предлагает меню:

1. **Создать задачу** – выбрать тип (file/cli/api), а потом решить: настраивать поля вручную или положиться на генератор источника.
2. **Сменить статус задачи** – ввести тип и ID задачи, новый статус. Реализовано с помощью `set_task_status` из `src/tasks_functions.py`.
3. **Забрать первые N задач из очереди** – они удаляются и печатаются на экран. Реализовано с помощью `get_left_tasks` из `src/tasks_functions.py`.

После каждого действия можно продолжить или выйти.


## Тесты

Для тестов используется `pytest`. Все тесты лежат в папке `tests/` и разделены по логике:

+ `conftest.py` – общие фикстуры: данные для задачи и очередь из 3 задач, чтобы не писать одно и то же в каждом файле.
+ `test_task_model.py` – проверка валидации полей (типы, длины, диапазоны, статусы), свойств `is_on_time` и `summary`.
+ `test_queue.py` – тесты очереди: добавление, итерация, фильтрация по условию/статусу/приоритету/дедлайну, поиск по ID, извлечение из пустой очереди.
+ `test_sources.py` – проверка, что каждый источник (FileSource, ConsoleSource, APISource) возвращает корректный объект `Task`.
+ `test_tasks_functions.py` – интеграционные тесты для `create_task`, `set_task_status`, `get_left_tasks` (с моками ввода, но для простоты без логгера).

### Запуск тестов

Из корня проекта (где лежит `pyproject.toml`):

```bash
pytest
```

## Покрытие кода

Для оценки покрытия используется pytest-cov:
```bash
# Просто покрытие
pytest --cov=src

# Покрытие + список непокрытых строк
pytest --cov=src --cov-report=term-missing

# HTML-отчёт (открыть htmlcov/index.html)
pytest --cov=src --cov-report=html
```

## Некоторые детали по коду
В tests/conftest.py вы можете наблюдать следующую строку: `status="created" if i != 3 else "finished"`. Кому-то такой однострочный if-else может не понравится, но такой выбор обусловлен 2 факторами. 1ое - Желая оставить изначальную идею, так писать было куда проще. 2ое - автор буквально в день написания тестов читал Rust Book, где в официальных примерах был такой шаблон(Python - не Rust, но всё же). Только поэтому я оставил такую строку.
